## 2. 前端UI需求概述

### 2.1 目标用户

根据requirements.md 2.1节，系统有两类主要用户：

#### 👨‍🌾 温室种植/生产管理人员（主要用户）

**核心需求**：
- 每日查看今天灌水量、预警信息、决策理由
- 可以Override系统建议，输入替代值和理由
- 不需要技术背景，界面简洁直观

**交互特点**：
- 移动端优先（温室现场操作）
- 大按钮、清晰文字、减少层级
- 中文界面、避免专业术语

#### 👩‍🔬 实验室研究人员（次要用户）

**核心需求**：
- 查看历史决策记录、趋势分析
- 批量导出数据用于论文
- 周度报告查看、Prompt调优

**交互特点**：
- PC端为主
- 数据密集型展示、支持筛选排序
- 支持参数调整、配置管理

### 2.2 设计原则

根据用户要求："科技感、不过于浮夸、不过于简单"

**色彩方案**：
- 主色：深蓝绿（#1E3A5F + #2ECC71 渐变）- 科技+农业
- 辅色：橙色预警（#E67E22）、灰色背景（#ECF0F1）
- 避免过于炫酷的霓虹色

**视觉风格**：
- 卡片式布局（Material Design 3.0风格）
- 圆角适中（8px）、阴影轻柔
- 图表使用柔和渐变、避免过多动画

**信息密度**：
- 核心指标突出显示（大号字体+数字）
- 次要信息可折叠收起
- 移动端一屏最多3个核心信息块

└─────────────────────────────────────────────────────────────┘

#### 关键组件

**1. 今日决策卡片（Hero Card）**

```json
{
  "数据来源": "GET /api/episodes/latest",
  "字段": {
    "date": "日期",
    "final_decision.value": "灌水量（大号数字）",
    "final_decision.source": "来源标签（TSMixer/Override）",
    "plant_response.trend": "长势趋势（better/same/worse + 图标）",
    "plant_response.confidence": "置信度",
    "sanity_check.risk_level": "风险等级（low/medium/high/critical）",
    "sanity_check.requires_confirmation": "是否需要确认"
  },
  "交互": {
    "查看详情": "跳转到 /daily/:date",
    "人工覆盖": "弹出Override对话框"
  }
}
```

**状态指示颜色**：
- 正常（low risk）：绿色边框
- 需关注（medium risk）：黄色边框
- 需确认（high risk）：橙色边框
- 紧急（critical risk）：红色边框 + 闪烁动画

**2. 灌水量趋势图**

- **图表类型**：折线图（Line Chart）
- **X轴**：日期
- **Y轴**：灌水量 (L/m²)
- **数据点**：
  - TSMixer预测：实心圆点
  - Override：空心菱形（橙色）
  - 长势better：绿色上箭头标注
  - 长势worse：红色下箭头标注
- **工具提示**：悬停显示完整信息
- **筛选**：支持7天/14天/30天切换

**3. 预警事项列表**

```
优先级排序：
  critical > high > medium > low

显示内容：
  • 风险类型 + 具体描述
  • 时间戳
  • [查看详情] 链接
```

### 4.3 人工覆盖（Override）对话框

**触发场景**：
1. 种植人员不同意系统建议
2. SanityCheck标记 `requires_confirmation: true`

#### 对话框布局

```
┌─────────────────────────────────────────────┐
│  ⚠️ 人工覆盖系统建议                          │
├─────────────────────────────────────────────┤
│  日期: 2025-12-26                          │
│  系统建议: 5.2 L/m²                        │
│                                            │
│  替代灌水量 (L/m²): [    6.0    ] ⬆⬇      │
│                                            │
│  覆盖理由: *                                │
│  ┌───────────────────────────────────────┐ │
│  │ [快速选择理由]                         │ │
│  │ • 高温预警，需增加水量                 │ │
│  │ • 观察到轻微萎蔫                       │ │
│  │ • 实验需要                             │ │
│  │ • 其他（请手动输入）                   │ │
│  └───────────────────────────────────────┘ │
│                                            │
│  ┌───────────────────────────────────────┐ │
│  │ (文本框，支持手动输入详细理由)         │ │
│  │                                        │ │
│  └───────────────────────────────────────┘ │
│                                            │
│  需要人工确认的问题 (来自SanityCheck):     │
│  • 今日湿度是否持续过高？                  │
│  • 是否观察到病害迹象？                    │
│                                            │
│  [取消]              [确认覆盖]            │
└─────────────────────────────────────────────┘
```

#### 数据提交

```json
POST /api/override

{
  "date": "2025-12-26",
  "original_value": 5.2,
  "replaced_value": 6.0,
  "reason": "高温预警，需增加水量",
  "confirmation_answers": {
    "今日湿度是否持续过高": "是，连续3天>80%",
    "是否观察到病害迹象": "否"
  }
}
```

#### 验证规则

- 替代值范围：[0.1, 20] L/m²
- 理由字段必填（至少10字）
- 提交成功后自动刷新Dashboard

## 5. 技术栈建议

### 5.1 前端框架

| 技术 | 推荐方案 | 备选方案 | 说明 |
|------|----------|----------|------|
| **框架** | React 18 + Vite | Next.js | 项目规模中等，Vite快速 |
| **UI库** | Material-UI (MUI) v5 | Ant Design | 卡片组件丰富、科技感 |
| **图表** | ECharts | Recharts | ECharts定制性强 |
| **状态管理** | Zustand | Redux Toolkit | Zustand轻量、易学 |
| **路由** | React Router v6 | - | 标准方案 |
| **请求库** | Axios + TanStack Query | SWR | Query缓存强大 |
| **表单** | React Hook Form | Formik | 性能好、验证灵活 |
| **类型检查** | TypeScript | - | 必须 |
| **样式方案** | MUI sx + Emotion | Tailwind CSS | MUI自带 |

### 5.2 后端API设计

**RESTful API端点**：

```
GET    /api/episodes/latest                # 获取最新Episode
GET    /api/episodes/:date                 # 获取指定日期Episode
GET    /api/episodes?start=2024-03-14&end=2024-06-14&trend=better  # 筛选查询
POST   /api/override                       # 提交Override
PUT    /api/episodes/:date/feedback        # 提交用户反馈

GET    /api/weekly                         # 获取所有周报
GET    /api/weekly/:week_start             # 获取指定周报

GET    /api/knowledge/search?q=开花期&top_k=5  # RAG检索
POST   /api/knowledge/feedback             # 检索反馈

GET    /api/images/:date/today             # 获取今日图像
GET    /api/images/:date/yesterday         # 获取昨日图像
GET    /api/images/:date/compare           # 获取对比图像

GET    /api/settings                       # 获取配置
PUT    /api/settings                       # 更新配置
```

**响应格式**：

```json
{
  "success": true,
  "data": { ... },
  "error": null,
  "timestamp": "2025-12-26T10:00:00Z"
}
```

**错误处理**：

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "EPISODE_NOT_FOUND",
    "message": "2024-12-30 的Episode不存在",
    "details": null
  },
  "timestamp": "2025-12-26T10:00:00Z"
}
```

### 5.3 部署建议

**前端**：
- 构建：Vite build
- 部署：Nginx静态托管
- CDN：可选（加速图像加载）

**后端API**：
- 框架：FastAPI (Python) - 已有Python后端代码
- ASGI服务器：Uvicorn
- 反向代理：Nginx

**架构示意**：

```
用户浏览器
    │
    ├── 静态资源请求 → Nginx (前端) → React SPA
    │
    └── API请求 → Nginx (反向代理) → Uvicorn (FastAPI)
                                        │
                                        ├─→ MongoDB (Episodes)
                                        ├─→ Milvus (RAG)
                                        └─→ LLM Service
```

## 8. 待讨论问题

### 8.1 核心问题

**Q1: 移动端 vs PC端优先级？**

- **选项A**: 移动端优先（种植人员现场操作）
- **选项B**: PC端优先（研究人员办公室）
- **推荐**: 移动端优先，PC端同步开发（响应式设计）

**Q2: 实时数据 vs 批量离线？**

- **选项A**: 实时流式更新（WebSocket）
- **选项B**: 定时轮询（每10s刷新）
- **选项C**: 手动刷新
- **推荐**: 定时轮询（简单可靠）

**Q3: 图像存储方式？**

- **选项A**: Base64嵌入JSON（单次请求）
- **选项B**: 图像CDN + URL引用（分离请求）
- **推荐**: 图像CDN（减少API负载）

**Q4: Override需要审批流程吗？**

- **选项A**: 直接提交即生效
- **选项B**: 提交后需管理员审批
- **推荐**: 直接生效（记录日志即可）

**Q5: 知识库支持用户上传文献吗？**

- **选项A**: 只读系统文献（FAO56）
- **选项B**: 支持用户上传（需审核）
- **推荐**: MVP阶段只读，Phase 3支持上传

### 8.2 UI细节讨论

**1. 颜色主题偏好**：
- 深蓝绿（科技+农业）
- 深紫色（科技感）
- 深灰+亮绿（现代简约）

**2. 图表交互**：
- 是否需要缩放（zoom）功能？
- 是否需要导出图表为图片？

**3. 预警通知方式**：
- 仅界面内提示
- 邮件通知
- 微信/钉钉推送
- 短信（紧急情况）

**4. 数据保留时长**：
- Episode保留多久？（建议永久）
- 图像保留多久？（建议1年）
